Designing a Scalable Enterprise HRMS Platform with ERP-Integrated Workflows

Problem Statement
Who was struggling, with what, and why it mattered

The organisation was running HR operations across a patchwork of disconnected tools — spreadsheets for payroll, email chains for approvals, separate systems for leave and attendance, and manual processes for asset tracking. No single source of truth existed. Every HR function operated in isolation, creating a compounding problem: errors multiplied across systems, audit trails were non-existent, and month-end processes were manual and stressful.

The users and their pain
HR Managers: no consolidated view of employee data across the lifecycle
Payroll Admins: manual deduction calculations, no automated leave integration
Employees: zero visibility on request status — no way to track leave, payslips, or approvals
Finance: duplicate data entry, no audit trail, compliance risk every quarter
IT Admins: asset allocation managed in spreadsheets with no ownership history
The systemic gaps
4+ disconnected tools with no data sync between them
Approval decisions made over email — no routing logic, no SLA, no escalation
Role-based access undefined — anyone with system access could see anything
No ERP integration readiness — payroll data was incompatible with finance systems
Zero design standards — every new feature was built independently

Key Design Decisions
The thinking behind the choices — not just what was built, but why
01
Dynamic role model — access as data, not code
+
Problem

Organisations change roles constantly — people get promoted, teams are restructured, new functions are created. Hardcoded permission logic means every role change requires a developer, a release, and a deployment cycle.

Decision

Design the entire role hierarchy as a data model — not as application logic. Roles, permissions, and access scopes are stored as configurable records, not as if-else conditions in code. Admins can modify access without engineering support.

Reasoning

This is how Google and enterprise SaaS platforms handle access at scale. The short-term cost is a more complex data model and admin UI. The long-term gain is a system that organisations can actually maintain without vendor dependency.

02
Payroll as a pipeline, not a form
+
Problem

Payroll admins had no visibility into where a payroll run was in the cycle. Was it still in validation? Had someone already approved it? The process was opaque and stressful every month-end.

Decision

Redesign the payroll interface as a 5-stage visual pipeline: Data Sync → Calculation → Validation → Approval → Disbursement. Each stage shows status, the responsible role, and what action is pending.

Reasoning

Payroll is a process, not a form submission. Visualising it as a pipeline reduces cognitive load, makes accountability explicit, and eliminates the need for status-update emails — the biggest source of payroll friction identified in discovery.

03
One approval engine for all 14 modules
+
Problem

Every module had its own implicit approval logic buried in email chains. Building module-specific approval UI would mean 14 different patterns, 14 inboxes, and zero shared audit trail.

Decision

Build one configurable approval engine that all 14 modules route through. Admins define routing rules, approval chains, escalation paths, and SLAs through a no-code drag-and-drop builder — without touching code.

Reasoning

Consistency over customisation. One engine means one inbox for managers, one audit log for compliance, and one set of UX patterns for users to learn. The configuration flexibility ensures each module's unique rules are still respected.

04
SAP-aligned data model from requirements phase
+
Problem

Most HR systems are built first, then integrated with ERP later — requiring expensive data transformation layers, field remapping, and months of integration rework.

Decision

Before designing any payroll UI, I authored field-mapping specifications in the FRD that aligned all salary components, deduction categories, and statutory fields with ERP schema. The UI was then built on top of an already-compatible data model.

Reasoning

Integration rework is one of the most expensive problems in enterprise software. Preventing it costs 2 hours of documentation. Fixing it costs weeks of engineering. This decision was made at requirements stage — not during development.

05
Design system first — screens second
+
Problem

With 14 modules and 250+ screens to design in 3 months, designing each screen independently would produce an inconsistent UI and make developer handoff painful and slow.

Decision

Build the design system and component library first — before any module screens. Every component (forms, tables, status badges, modals, approval cards) was standardised and reusable before a single module was designed.

Reasoning

At enterprise scale, consistency IS usability. Users shouldn't have to learn a new interface pattern for every module. Building the system first also reduced design time by ~40% and cut developer questions about component behaviour dramatically.

06
Visibility over speed in the employee experience
+
Problem

Employee frustration with HR systems isn't usually about speed — it's about not knowing what's happening. "Where is my leave request?" and "Has my payslip been processed?" are the most common HR helpdesk queries.

Decision

Design the employee-facing interface around status transparency — not just task completion. Every request has a real-time 4-step timeline. The dashboard surfaces deadline-driven actions contextually. Nothing requires a helpdesk call to understand.

Reasoning

Reducing helpdesk queries is a measurable business outcome. Transparency reduces anxiety. This was validated through informal walkthroughs — employees consistently cited "not knowing status" as their biggest frustration with the previous system.

Deep Dive — The Approval Workflow System
Step-by-step flow, edge cases, and complexity handled

The approval system is the most complex module in the platform — it is the connective tissue between every other module. Any request generated anywhere in the system — leave, payroll changes, asset allocation, hiring requisitions, expense claims — flows through this engine. I chose to deep-dive this module because it best demonstrates the intersection of system thinking, UX design, and BA work.

1
Request initiation — employee submits any request type
An employee submits a request (e.g. leave application). The system identifies the request type, maps it to the appropriate approval chain via the routing rules engine, and generates a unique request record with full metadata.
Edge case: What if the employee's direct manager is also on leave? The system auto-escalates to the next tier in the hierarchy — no manual intervention needed.
2
Routing engine — dynamic chain resolution
The routing engine reads the approval matrix (configured by the HR Admin) and determines which roles need to approve, in what sequence, and with what SLA. This is pure data — no hardcoded logic.
Edge case: What if an approver has already been assigned a delegation? The engine resolves delegation chains before routing — primary → delegate → next tier, in order.
3
Approver experience — unified inbox with context
The approver sees all pending requests in one inbox, filtered by module, urgency, and SLA status. Each request shows full context inline — no need to navigate to another module to understand the request. Approve, reject, or delegate in one click.
Edge case: What if the SLA is breached? The system sends escalation notifications and flags the request visually in the inbox — the approver cannot miss it.
4
Requester experience — status transparency
The requester sees a 4-step timeline (Submitted → In Review → Decision → Actioned) with a timestamp at each stage. They can see who is currently reviewing their request. No helpdesk call required.
5
Audit log — immutable decision record
Every decision — who approved, at what tier, at what timestamp, with what rationale — is written to an immutable audit log. This was a non-negotiable compliance requirement identified in the BRD. It cannot be edited or deleted by any user role, including Super Admin.
Edge case: What if a decision needs to be reversed? Reversals create a new record (a reversal event) that references the original — the original decision is preserved. Full trail maintained.

Challenges & Trade-offs
Where the hard decisions were and what was traded away
⚖
Flexibility vs simplicity in the approval engine: The configurable chain builder gives HR admins maximum flexibility — but it also means a more complex admin interface. We traded UI simplicity for admin power, accepting that the admin user is a sophisticated HR professional who needs the control, not a casual end user.
🔗
ERP integration constraints shaped UI decisions: Several UI patterns I initially designed had to be revised when I mapped the data against ERP schema. Some field combinations that made UX sense didn't align with how the ERP expected data. Learning to treat ERP constraints as design constraints (not just dev constraints) was a significant shift in how I approached the requirements phase.
⏱
3 months for 250+ screens — real time pressure: The design system decision was partly a necessity. Without it, delivering 250+ consistent screens in 3 months would have been impossible. The trade-off was spending the first 2 sprints building the system rather than screens — which required stakeholder trust in a slower start for a faster delivery.
👥
Designing for 9 roles with conflicting needs: What is the right dashboard for a Payroll Admin is wrong for an Employee, and both are wrong for a Super Admin. Managing these conflicting requirements without creating 9 entirely separate products required strong design principles and a shared component system that could be composed differently per role.
K
Learnings
What this project changed about how I think
01
Enterprise UX is system design, not screen design
The quality of an enterprise product is determined by data models, integration points, and role architecture — not by the visual polish of individual screens. I entered this project thinking like a designer. I left thinking like a system architect.
02
Requirements are design decisions
Every BRD requirement I wrote shaped what was possible in the UI. Defining the approval matrix correctly in the FRD prevented months of design rework. The BA and UX work are not sequential — they are simultaneous.
03
Constraints are your best design tool
ERP integration constraints, role hierarchy limits, and sprint timelines all forced better decisions. Unconstrained design produces endless options. Constrained design produces clarity.
04
The dual role is a superpower
Having one person who understands both business requirements and UX design eliminates the most expensive friction in product development — the translation gap between what the business asked for and what gets designed. I intend to keep operating this way.
